# Claude 长时间回复说明

这套桥接方式里，Claude 有时会因为任务复杂而回复很慢，尤其是这些情况：

- 审图
- 审代码
- 读多个文件
- 总结长日志
- 需要多步推理的分析任务

出现几分钟甚至十几分钟的等待，不一定代表失败。

## 当前建议

1. 把 Claude 侧边栏保持开着。
2. 如果终端里还在转圈、还在输出状态、或者还没有回到提示符，就把它当成“还在工作”。
3. 不要因为几分钟没出最终答案就立刻重发。
4. 如果任务确实很大，允许等待更久，最好按 5 到 10 秒轮询一次输出。
5. 如果用户想中断，再发送 `Ctrl+C`。

## 这套 skill 的预期行为

- Codex 把消息发给 Claude 后，不会因为短时间没答完就立刻放弃。
- 对于长任务，应继续轮询 Claude 输出。
- 在没有明显失败信号的情况下，可以等待到大约 20 分钟左右，再把它视为可能卡住。
- 如果等待时间较长，Codex 应给用户一个简短更新，然后继续等，而不是直接结束这次交换。

## 怎样判断是真的卡住了

更像卡住的信号通常是：

- Claude 已经回到提示符，但没有真正回答任务。
- 终端长时间完全没有任何新增输出，而且不是在正常思考状态。
- Claude 进程退出了。
- 用户明确要求停止。

如果只是慢，不等于失败。
