# Claude Relay 风险和处理建议

这套桥接里，最常见的三个风险点是：

## 0. Codex 以为已经发给 Claude 了，但其实还没真正送进去

建议处理方式：

- 先记录发送前的 `cursor`。
- 调用发送接口后，先确认 `POST /api/input` 返回成功。
- 然后立刻再看一次新的终端活动，确认消息已经真正进入 Claude 当前会话。
- 只有确认“已经发出去”之后，才进入“等待 Claude 回复”的阶段。

可以接受的发送确认信号包括：

- `ok: true`
- `wrote` 大于 0
- `cursor` 前进了
- 终端里出现了新的输出
- Claude 出现了 spinner、状态文本、trust prompt 或消息回显

如果没有这些信号，就不要直接等回复，而是先检查有没有 trust prompt、阻塞状态或者发送失败。

## 1. Codex 发给 Claude 的内容太长，或者中文内容在终端里不稳

建议处理方式：

- 能用文件路径就不用大段粘贴。
- 如果内容已经在共享工作目录里，直接把路径发给 Claude，并明确说“请打开这个文件并检查”。
- 如果内容还不在文件里，而且很长，先写成临时文件，再让 Claude 去读这个文件。
- 中文或中英混合内容也优先走文件路径，而不是长段终端粘贴。

原因：

- 终端粘贴虽然通常可用，但长内容和多语言内容更适合文件直读。
- 文件路径方式更稳定，也更利于 Claude 反复查看上下文。

## 2. Claude 回给 Codex 的内容太长，终端缓冲可能不够

建议处理方式：

- 对于可能很长的审查结果，不要只让 Claude 在终端里直接输出全部内容。
- 更稳的做法是让 Claude 把完整结果写到共享工作目录里的某个文件，然后在终端只回复简短摘要和文件路径。
- Codex 再直接读取那个输出文件。

推荐提示词写法：

```text
请把完整审查结果写到 D:\your-project\claude-review.md，然后在终端里只给我一个简短总结和这个文件路径。
```

## 3. Claude 回复很慢，几分钟甚至十几分钟

建议处理方式：

- 不要因为几分钟没出最终答案就直接判断失败。
- 如果 Claude 还在输出状态、还没回到提示符、或者还有增量输出，就把它视为仍在工作。
- 对于复杂任务，允许更长等待时间。
- 如果需要，可以给用户一个简短进度更新后继续等。

## 当前 skill 已经覆盖到的规则

当前 `shared-claude-bridge` skill 已经要求：

- 默认发送后立即提交，不留草稿。
- 本地文件、图片、截图优先传路径并明确要求 Claude 去读。
- 长内容优先转成文件再交给 Claude。
- 长回复优先让 Claude 落盘，再给终端摘要。
- 慢任务允许持续轮询等待，不会因为几分钟没回复就立刻放弃。

## 结论

第 1 和第 2 个问题，大部分都可以通过 skill 约束和提示词写法明显降低风险。

第 3 个问题，也已经可以通过 skill 的等待策略来规避大多数“等不及”的情况。

但要注意：

- 真正的文件编码问题，不能只靠 skill 完全消灭。
- 超长终端输出的根本限制，除了 skill 规避之外，也可以结合后端缓冲上限调整来进一步增强。
